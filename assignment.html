OpenGL Math Library

In the C++ introduction assignment you implemented a matrix and vector class. In this assignment we will use the glm library to do this for us. glm provides 2D, 3D and 4D vectors and matrices as well as many operations that can be applied to these types.

We can create a 2D/3D/4D glm vector in various ways:

glm::vec3 myFirstVec3 { 1, 2, 3 }; // Creates a vector (1, 2, 3)
glm::vec3 singleValueVec3 { 5 }; // Creates a vector (5, 5, 5)

glm::vec2 multiValueVec2 { 1, 2 }; // Creates a vector (1, 2)
glm::vec3 multiValueVec3 { myFirstVec2, 3 }; // Creates a vector (1, 2, 3)
glm::vec4 multiValueVec4 { multiValueVec3, 4 }; // Creates a vector (1, 2, 3, 4)

The glm library makes use of operator overloading. Operator overloading is a powerfull feature in c++ which allows us (or in this case glm) to define the behavior of the +, -, * and / operands for a custom class. In practice, this means that math heavy code is greatly simplified.

glm::vec3 a { 2, 5, 3 };
glm::vec3 b { 9, -3, -1 };
glm::vec3 c = a + b; // (11, 2, 2)

glm provides functions to create transformations such as rotation, translation and scale. These functions take an existing matrix, compute the desired transformation matrix and multiply it by the existing matrix. The multiplication order is such that the transform operation is performed before the transformation stored in the input matrix.

// Create an identity matrix (1 on the diagonal, 0 everywhere else).
glm::mat4 identity = glm::identity<glm::mat4>();

float angle = glm::radians(45.0f); // Convert angle from degrees to radians.
glm::vec3 axis { 0, 1, 0 }; // Positive y-axis
// Rotate around the positive y-axis by 45 degrees
glm::mat4 rotation = glm::rotate(identity, angle, axis);

// Translate by (1, 2, 3) *then* rotate around the positive y-axis by 45 degrees.
glm::mat4 translateThenRotate = glm::translate(rotation, glm::vec3(1, 2, 3));

// Scale by (1, 2, 3) *then* rotate around the positive y-axis by 45 degrees.
glm::mat4 scaleThenRotate = glm::scale(rotation, glm::vec3(1, 2, 3));

// Multiply the vector (0, 1, 2, 1) in homogenous coordinates by the scale/rotate matrix.
glm::vec4 resultHomogeneous = scaleThenRotate * glm::vec4(0, 1, 2, 1);
// Convert from homogenous coordinates back to cartesian coordinates.
glm::vec3 result = glm::vec3(resultHomogeneous) / resultHomogeneous.w;

We provide a couple of helper functions which compute the rotation / translation / scale matrices directly. Using these functions is optional but it allows you to write matrix multiplications in right to left order which is common in mathematical literature.

glm::mat4 rotationMatrix(float angle, const glm::vec3& axis);
glm::mat4 translationMatrix(const glm::vec3& translation);
glm::mat4 scaleMatrix(const glm::vec3& scale);

float angle = glm::radians(45.0f); // Convert angle from degrees to radians.
glm::vec3 axis { 0, 1, 0 }; // Positive y-axis
glm::mat4 translateThenRotate = rotationMatrix(angle, axis) * translationMatrix(glm::vec3(1, 2, 3));

2. Solar System

For this second exercise we will “simulate” a solar system. For the sake of simplicity we will only consider rotations around a single axis (the positive y-axis glm::vec3(0, 1, 0)). A solar system consists of various celestrial bodies (spheres) which orbits around one another. The one exception to that is the sun (star) which is at the origin of our solar system (glm::vec3(0)). Apart from orbiting, a celestrial body also spins around it’s own axis. Finally, each body may have a different size (radius).

Each celestrial body is represented by the following struct:

struct CelestrialBody {
    // The radius of the sphere that represents the celestial body.
    float radius { 1.0f };
    // A celestial body may spin around it's own axis. The speed at which it rotates is
    //  expressed as the spin period. The spin period is the amount of time (in hours) it
    // takes for the body to complete one full rotation.
    float spinPeriod { 0.0f };

    // A celestial body may orbit another celestial body (e.g. a moon orbiting a planet).
    // orbitAround stores the index of the object inside the bodies collection that this
    // celestial body is orbiting. The altitude describes the distance at which the object
    // is orbited (as measured from the center of both objects). The orbit period describes
    // how much time it takes for the celestial body to complete one rotation around the
    // object being orbited.
    //
    // If orbitAround is negative then this celestial body is not orbiting anything and it
    // should instead be placed at the origin (and orbitAltitude/orbitVelocity may be ignored).
    int orbitAround { -1 };
    float orbitAltitude { 0.0f };
    float orbitPeriod { 0.0f };
};

Each body is rendered as a sphere centered at the origin with radius=1 which is transformed by the matrix that you return. The provided implementation returns an identity matrix for each celestrial body, hence they are all drawn at the origin with the same radius.

You need to implement the following function in solar_system/src/your_code_here.h. It should return for each celestrial body a matrix that describes the transformation at the given time. This transformation matrix should rotate / translate / scale the input sphere (radius=1, center=(0,0,0)) according to the CelestrialBody struct.

std::vector<glm::mat4> computeCelestrialBodyTransformations(
    gsl::span<const CelestrialBody> celestrialBodies, float time);

Both the orbit and spin should use the positive y-axis (glm::vec3(0, 1, 0)) as the axis of rotation. Both the orbit and spin are described using a period which is the amount of time it takes for the body to complete a full rotation (2.0f * glm::pi<float>()). At time=0 each body should be to the right (positive X-axis) of the celestrial body that it is orbitting around.

Orbit and spinning are two separate operations that should not impact each other in any way:

  • Say we have a planet A spinning around it’s own axis every 24 hours. A geo-stationary satalite orbitting around the planet will have a orbit period of 24 hours as well (and not 0!).

  • The same holds true for “nested” orbitting (e.g. moon orbits planet orbits sun). If the moon has an orbit speed of zero (hypothetical infinite period) then it would always stay on the same side of the planet as seen from a static point in space, such as the camera. Thus, the moon would create a solar eclipse on the half way point of the planet’s orbit around the sun.

  • The orbit of a body around another one should not impact it’s own spin. So if the spin speed were zero then the orientation of the body should not change relative to a static observer while it is orbitting. This is illustrated in the image below.
